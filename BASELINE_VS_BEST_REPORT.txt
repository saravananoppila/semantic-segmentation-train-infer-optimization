================================================================================
Date:      2026-06-29
Hardware:  NVIDIA L4 (23,034 MiB), 4 vCPUs | torch 2.4.1+cu121
Model:     HRNetV2 + C1 decoder, 150 classes, ADE20K
Schedule:  identical for both — 10 epochs x 1000 iters = 10,000 optimizer steps,
           seed 304, then eval on 2000 validation images.

READ THE TWO COMPARISON AXES SEPARATELY:
  * Training / GPU metrics are a CLEAN A/B — same 10k-step schedule, same
    hardware, only the config differs.
  * Model-quality metrics (mIoU / acc) are NOT data-matched. Because batch=11 vs
    batch=2 over the same step count, the best config sees 110k image-
    presentations vs the baseline's 20k (5.5x more). The mIoU gap therefore
    reflects the optimizations AND the larger data exposure the bigger batch
    buys. See section 5 for the honest interpretation.

 5. MODEL-PERFORMANCE METRICS   (NOT data-matched — interpret carefully)
--------------------------------------------------------------------------------
  Metric                   Baseline      Best config
  -----------------------  -----------   -----------
  Mean IoU                 0.1431        0.3486
  Pixel accuracy (val)     69.33%        78.67%
  Per-class IoU median     0.038         0.341
  Zero-IoU classes         ~49 / 150     3 / 150
  Final train pixel-acc    71.85%        84.74%
  Final train loss         1.110         0.515
  Inference time           0.270 s/img   0.268 s/img

  Honest reading: at an EQUAL STEP budget (10k iters), the best config produces a
  much stronger model (2.4x the mIoU). But that is NOT "same data, better model"
  — the batch=11 config saw 5.5x more images, and that extra exposure is a large
  part of the gap. A data-matched baseline (55k iters @ batch 2 = same 110k
  images) would close much of the mIoU difference. Inference latency is identical
  (same architecture; precision affects training, not this fp32 eval path).

  So the model-quality numbers answer "in the same number of training steps,
  which config gives a better model?" (best config, decisively) — not "per image
  seen, which is better?" (roughly equal).

--------------------------------------------------------------------------------
 7. BOTTOM LINE
--------------------------------------------------------------------------------
  * Speed: the optimized stack is a clean 1.94x throughput win at the same step
    budget, same hardware, no instability — convergence run confirms it doesn't
    break training.
  * Memory: the baseline wastes 68% of the GPU. The single most important
    optimization is spending that memory on a larger batch (OFAT's biggest
    lever, +43%).
  * Model quality: better at equal steps, ~comparable per-image. The
    optimizations don't degrade the model; the mIoU lead at equal steps comes
    largely from the extra data the big batch processes.
  * For a publishable accuracy comparison, re-run both data-matched (equal
    image-presentations); for a throughput/efficiency comparison, this report is
    already apples-to-apples.
