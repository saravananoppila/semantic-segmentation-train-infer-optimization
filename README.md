# HRNetV2 Semantic Segmentation — Training & Inference Performance

This is my personal practice project. I took an open-source HRNetV2 + C1 semantic
segmentation model (ADE20K) and used it as a hands-on sandbox to learn how to
**profile and improve a real deep-learning training and inference pipeline** on GPUs.

The focus here is **not** the model itself but the **performance engineering** around
it: measuring where time actually goes with Nsight Systems and `nvidia-smi` telemetry,
then testing optimizations one at a time and keeping only the ones that measurably help.

---

## Goal

Practise the full performance-optimization workflow end to end:

1. **Profile** the baseline to find the real bottleneck (not the assumed one).
2. **Optimize training throughput** — get more images/second without hurting accuracy.
3. **Optimize inference latency** — make a trained model serve faster and cheaper.
4. **Validate** every change: speed *and* model quality, with reproducible runs.

---

## Training results (NVIDIA L4)

Systematic One-Factor-At-A-Time (OFAT) study; each run profiled with nsys + 1 Hz telemetry.

| | Baseline | Optimized | Change |
|---|---|---|---|
| Throughput | 7.93 img/s | **16.86 img/s** | **1.94×** |
| GPU utilization | 90.6% | 98.5% | GPU kept saturated |
| GPU memory used | 32.4% | 91.4% | idle VRAM turned into work |
| Mean IoU (proxy run) | — | 0.349 | convergence validated, stable |

**What worked** (the workload is compute-bound, so the wins were "fill the GPU" + "better kernels"):
- Max batch size 2 → 11 — **+43%** (fills the idle 68% of VRAM)
- `channels_last` (NHWC) — **+23%** (tensor-core-friendly conv kernels)
- AMP → **BF16** (halves activation memory, enables the bigger batch)
- Fused SGD + fused loss — small, free wins

**What I tried and rejected** (all regressed or were neutral on this compute-bound net):
`cudnn.benchmark`, `torch.compile`, nvJPEG GPU decode, DALI training pipeline,
gradient checkpointing, TF32, grad accumulation. Knowing what *doesn't* help was as
useful as what does.

## Inference results (NVIDIA T4)

Optimized a trained checkpoint for latency across several backends.

| Technique | Speedup vs FP32 | Accuracy | Note |
|---|---|---|---|
| **FP16 + TensorRT** | **3.1×** | no loss | recommended — best speed with zero accuracy cost |
| INT8 + TensorRT | 4.0× | −0.034 mIoU | fastest, smallest engine (95 MiB) |
| INT8 (CPU) | — | small loss | 4× smaller model, CPU-only serving |
| torch.compile (FP16) | +27% | no loss | best non-TensorRT GPU option |
| Conv+BN fold (eager) | +4% FP32 / +20% FP16 | no loss | pure PyTorch; subsumed by TensorRT's auto-fusion |
| OpenVINO / ONNX Runtime (CPU) | 2.16× / 1.54× | no loss | CPU-runtime comparison |
| DALI GPU preprocessing | ~38× vs CPU | — | removes the CPU decode/resize bottleneck (~4550 img/s) |

Also includes a Triton Inference Server deployment (`triton_deploy/`).

## Reproduce it

Full step-by-step recipe — environment, dataset, and the exact command for every
experiment — is in **[REPRODUCE.md](REPRODUCE.md)**.

Quick version:
```bash
python3.8 -m venv .venv && source .venv/bin/activate
pip install torch==2.4.1 torchvision==0.19.1 --index-url https://download.pytorch.org/whl/cu121
pip install -r requirements.txt                 # training
pip install -r requirements-inference.txt       # optional: TensorRT/DALI/ONNX/OpenVINO/Triton

./download_ADE20K.sh          # dataset
./run_convergence.sh          # best-config training + eval
./run_exp3_trt.sh             # TensorRT FP16/INT8 inference
```
